# 論文重新定位：便宜商用 WiFi（ESP32/52 子載波）＋ 演算法貢獻

> 目的：在「實驗改以 ESP32/52 子載波為主、AX211/256 留作 scaling 對照」的前提下，把論文的
> **主張、貢獻、範圍**釘清楚，讓 Abstract / Intro / Contributions / Scope 章節照這個骨架填。
> 定位原則：**子載波數是硬體參數，不是貢獻；貢獻是演算法、系統與實證發現，而且它們在受限硬體下更有價值。**

## 1. 一句話主張（Thesis）

> 把**取樣均勻性**做對，加上**姿態感知子載波選擇**與**場域校準**，最便宜的商用 WiFi
> （ESP32、52 子載波、善用既有 AP）就能做**臨床相關的單人呼吸／呼吸中止監測**；更多子載波
> （256/AX211）的價值在**多人、穿牆、定位**等難場景，屬未來高階層。

## 2. 核心賣點（最強、且為真實實驗掙得）

> **對商用 AP 的 CSI 感測而言，「取樣的均勻性」是一階決定因素，常比子載波數、比標稱採樣率更重要。**

- **實證 A/B**：乾淨**均勻** 6 Hz / 52 子載波的捕捉，勝過**爆發式** 111–125 Hz 的捕捉（後者中位數看似高，
  有效率只有 ~64 Hz、Δt cv≈1.3，估頻時呼吸峰塌到頻帶邊緣）。見
  [`real_data/second_batch_20260722/`](real_data/second_batch_20260722/) 與研究紀錄 §2.15。
- **意義**：翻轉「要昂貴硬體才行」的預設 →「把取樣做對，便宜硬體就夠」。又新、又實用、又是自己量出來的。
- **可落地**：`realdata.py` 的 `_sampling_stats()` 會自動偵測 bursty 並提示 `resample_uniform()`。

## 4. 52 vs 256 的處理：把 256 從主角降為「scaling 特性軸」

- **主實驗**：ESP32 / 52 子載波**真實資料**（單人呼吸、呼吸中止），走均勻取樣。
- **256 / AX211**：用**數位孿生模擬**（或借用公開 AX211 資料）畫「**子載波數 vs 效能**」曲線，
  論證**核心任務在 52 已足**，而 256 買到的是**難場景（多人 / 穿牆 / 定位）**的餘裕。
- 好處：256 變成「52 夠用」的**對照證據**，同時點出未來高階版方向——誠實、且一魚兩吃。

## 5. 範圍聲明（Scope，主動寫進論文避免被質疑）

- **已驗證主張**：**單人**呼吸／呼吸中止，室內、人在 AP–感測器連線附近。
- **明列限制 / future work**：
  - **多人分離、複雜動作**：單天線 20 MHz 的 ESP32 先天受限；真正的槓桿是**天線數(MIMO/AoA)＋頻寬＋演算法**，非僅子載波數 → 列為需要更豐富硬體的未來方向。
  - **穿牆 / 定位**：同上，屬高階層。
  - **真實 ground-truth**：呼吸率需碼表/金標準同步（見 EXPERIMENT_PROTOCOL.md）。
